Chrome Voice-Assisted AI Extension for the Blind
Building an AI-driven voice assistant as a Chrome extension involves leveraging the browser’s accessibility and extension APIs, advanced speech and vision models, and careful handling of security. At a high level, the extension will listen to spoken commands (via on-device speech recognition) and respond with synthesized speech, while using a multimodal AI (LLM plus vision) to understand page content and perform tasks. Core components include the Web Speech API (for voice input/output), Chrome extension APIs (content scripts, tabs, credentials, etc.), and a backend AI model (either cloud-based or open-source local) for understanding and memory. The assistant should manage a conversation “memory” (e.g. user preferences and past interactions) and handle sensitive actions like auto‐login securely. Below we outline the APIs, AI models, features, security considerations, and an implementation roadmap for this project.
1. Chrome Extension & Accessibility APIs
•	Manifest V3 Extension Framework: Use Chrome’s Manifest V3 model to define permissions (microphone, storage, activeTab, scripting, etc.) and background service workers. The extension will include content scripts injected into pages for DOM access and a background script for global orchestration.
•	Web Speech API (on-device): Chrome 139+ supports an on-device speech recognition mode via the standard Web Speech API (window.SpeechRecognition). This lets the extension capture microphone input and transcribe voice locally without sending audio to servers[1]. This on-device mode dramatically reduces latency and preserves privacy (no audio leaves the user’s machine[1]). The extension should request microphone permission early and use SpeechRecognition (or webkitSpeechRecognition) with continuous listening when voice mode is active[2][3].
•	Speech Synthesis / TTS: For output, use Chrome’s built-in Text-to-Speech (TTS) API (chrome.tts.speak() or the Web Speech SpeechSynthesis). This can read selected text or structured content aloud. Chrome’s TTS API supports multiple languages and voices, which can be controlled (rate, pitch) programmatically[4]. The assistant will use TTS to read page content, email text, or AI-generated responses back to the user.
•	Content Scripts & DOM Access: Content scripts will traverse the page DOM to identify meaningful text (headings, paragraphs, inputs) and interface elements (buttons, links, form fields). For example, to read aloud what’s on-screen, the script can query document.body.innerText or use ARIA roles and landmarks to segment text (e.g. headings vs. links). It can highlight or focus elements for easier navigation. For acting on commands (like “click the next email”), the script can find and click the corresponding DOM element (Gmail’s “Next” button, etc.).
•	Chrome Accessibility Features (if needed): Chrome provides a chrome.accessibilityFeatures API to toggle OS/browser accessibility settings (e.g. caret highlighting). However, for a voice-driven assistant, this is optional – the extension can itself manage focus and reading without changing system features.
2. Voice Interaction Design
•	Hotword / Activation: The assistant could be always listening once enabled, or activated by a hotkey/voice phrase. Care must be taken to avoid mis-triggering. A common pattern is to have the user press a key or click an icon to start listening (“push-to-talk”), then stop after the command is recognized (since background listening can be error-prone).
•	Speech Recognition Handling: As Roman Fedytskyi (Chrome dev) explained, developers should handle interim results and final results. The extension can display interim text for user confirmation and only act on final results[2]. Errors (no speech, misrecognition) should prompt a retry. Use continuous listening to parse multi-part commands (“open Gmail and read the first email”).
•	Natural Language Commands: When speech is transcribed, the assistant interprets it via an LLM. Commands can include browser actions (“Open new tab”, “Scroll down”, “Go back”), data tasks (“Read email”, “Write email to [name]”), and queries (“What is on screen?”). The extension should parse intent – either via simple command mapping (e.g. keywords) or using the AI model to classify commands.
•	Speech Output: The assistant should speak responses and content using TTS. For example, if the user says “read this page”, the assistant will synthesize the visible text. If the assistant generates an answer (“Here is the summary...”), it will speak that. Voice style and pace should be user-configurable (faster/slower, male/female voice).
3. Vision and Multimodal AI for Screen Understanding
•	LLM with Vision (Multimodal): To truly understand screen content (images, UI elements, formatted documents), we need a multimodal AI model. By 2026, open-source Vision-Language Models (VLMs) can accept screenshots or image inputs alongside text. For example, OpenAI’s GPT-4 (and GPT-4o) are multimodal: they accept images and text as input[5][6]. OpenAI’s model “exhibits similar capabilities” on text+image as on text alone, handling documents, screenshots, and diagrams[6]. However, using a cloud model raises privacy concerns (sending personal screen data externally).
•	Open-Source Multimodal Models: Alternatively, open-source VLMs allow local or self-hosted inference. Models like Gemma 3, Qwen-VL, Pixtral, Phi-4V, GLM-4.6V, and others now offer high-quality image-text understanding. For instance, Qwen3-VL can recognize UI elements, understand on-screen text, and even operate graphical interfaces via tool calls[7]. Models like DeepSeek-OCR specialize in accurate optical character recognition and page structure parsing[8]. These models can take a screenshot of the webpage (via the extension’s tabs.captureVisibleTab) and feed it to the VLM, which returns a structured description or answers queries (“This is a Gmail inbox with 10 unread emails”).
•	Use Cases: The vision-AI can enable features like: identifying buttons and fields (to click or fill), reading text from images or complex layouts (OCR for scanned PDFs), summarizing page content, and answering “What’s on my screen?” queries. For example, the user could say “Describe this chart”, and the assistant’s VLM could caption the image and highlight numeric trends. Or “What’s the title of this document?” by scanning the top of the page.
•	Latency and Privacy: Running multimodal models locally on-device may be heavy; realistically a server (possibly user-hosted on their own machine) or cloud is needed. If privacy is paramount, the assistant should default to on-device speech and optionally local VLM inference (e.g. running an optimized model on a desktop). Alternatively, encryption or on-premise servers can mitigate sending data externally. As an intermediate solution, low-level content (plain text DOM) can be handled by the extension without sending, and only images or private documents sent if the user consents.
4. Features & Interaction Flows
•	Voice Navigation: Basic browser navigation by voice: “Open new tab”, “Close this tab”, “Go back”, “Scroll down”, “Click that” (pointing via voice if needed), “Switch to tab [name]”. These map to chrome.tabs or content-script actions.
•	Reading Content Aloud: “Read this page” or “Read from here” – the extension takes visible text (or uses the content script to fetch meaningful chunks) and calls TTS to speak it. It should honor HTML structure (read headings first, then paragraphs) and allow pausing/resuming reading. The assistant can announce contexts (“At the top: Heading ‘Inbox’” or “Table of rows detected”).
•	Email Interaction (Gmail/Outlook): For Gmail or other web email: commands like “Open Gmail” (open tab to mail client), “List emails” (extension scrapes the inbox list and reads out senders/subjects), “Read email number 2” (reads the content of the second email), “Reply to this email” (invokes compose with LLM-generated suggestion), or “Compose new email to [name or address], subject X, body Y.” The extension can use Gmail’s DOM structure (or Gmail API if authorized) to perform these actions.
•	Documents/Spreadsheets: In Google Docs/Sheets or Office web apps: voice commands like “Open document X”, “Create new document”, “Go to cell B5”, “Write this text”, “Find [word]”, and “read cell range A1:C3”. The assistant can interface with the page’s editing API (in Docs) or simulate keystrokes/clicks via scripting. For writing, it can accept dictation and possibly refine it via LLM (“make it more formal”).
•	Form Filling & Password Login: The assistant can help fill forms. On a login page, a user could say “log me in to [site]”. The extension locates username/password fields and either uses stored credentials or asks the user (via voice prompt) for them. Ideally, integrate with Chrome’s credential manager: using navigator.credentials.get({password: true}) triggers Chrome’s account chooser[9] to select a saved login. Once chosen, the extension auto-fills the fields and submits. (Note: this requires user consent each time, as extensions cannot silently extract passwords.) Alternatively, the extension could store an encrypted credential in chrome.storage.local protected by a user PIN or OS biometric; it would then inject these values into the form fields when requested. (This approach must be done carefully and explicitly to avoid a security vulnerability.)
•	Conversations & Memory: The assistant maintains context. For example, if the user says “What’s my agenda tomorrow?” and the assistant knows the user’s calendar context, it can answer. A simple memory model can be built with an IndexedDB or chrome.storage database storing facts (preferences, past commands, recent pages). More ambitiously, use Retrieval-Augmented Generation (RAG): key points from past interactions are summarized and appended to prompts to the LLM. For instance, the assistant might “remember” that the user likes short summaries or that today’s task is to write a report, and use that memory when generating responses. Libraries like LangChain or similar can manage conversation buffers or vector stores for memory.
•	Error Handling: Always allow the user to say “undo” or “repeat” if the assistant misunderstood. Provide confirmations for destructive actions (“Sure you want to delete this email?”). Display a log or visual cues if needed (perhaps a small popup panel showing transcribed commands and results, mainly for debugging or sighted supervisors).
5. Privacy, Security, and Permissions
•	Data Privacy: All sensitive content (emails, private docs, screenshots) should be handled securely. Where possible, keep processing local: on-device speech recognition and local VLM inference mean no raw voice or image data leaves the machine[1]. If using cloud AI (OpenAI API or others), inform the user that data is being sent, and ideally allow an opt-out. In any case, the extension should not store unencrypted personal content. Stored memory should avoid plaintext sensitive info (passwords, private messages) unless encrypted with a user key.
•	Permissions: The extension will require broad host permissions (e.g. access to mail and docs URLs) and powerful APIs. Be mindful of Chrome’s extension policies: requesting too many permissions may scare users or get the extension flagged. A possible design is to request only needed permissions at runtime (using the optional permissions API) – for example, only ask for gmail.com access when the user first opens Gmail.
•	Auto-Login Security: Handling passwords is tricky. As noted, using the Credential Management API triggers a UI prompt rather than giving direct access[9]. If the assistant “sees” a login screen, it should ideally use Chrome’s own autofill (via navigator.credentials) so the user’s saved credentials fill in normally. If implementing its own storage, encrypt the credentials with a passphrase or use the browser’s Identity/Credential storage. Never log or transmit passwords over the network. For further safety, require explicit voice confirmations before sensitive actions (e.g. “Do you want to proceed with login on Gmail using saved credentials?”).
•	Secure Communication: If the extension communicates with a backend or LLM API, use HTTPS and API keys securely. Do not hard-code secrets; use Chrome’s chrome.storage or the secrets API to store keys. Ensure any local server (for a local model) uses local host or secure channels.
6. Feature Summary (Candidate List)
User Commands: (sample voice commands and assistant actions)
•	Navigation/Controls: “Go to Google,” “Open [site],” “Click [button/link],” “Find [text] on this page.”
•	Reading Content: “Read this page,” “Read [element]”, “Summarize this page.”
•	Email: “List my unread emails,” “Read the next email,” “Compose email to [Name],” “Reply to this email,” “Search emails for [keyword].”
•	Documents/Spreadsheet: “New document,” “Open [Doc name],” “Find [term] in this document,” “Insert table,” “Sum this column.”
•	Form Filling/Login: “Log in to [site],” “Fill [field] with [value],” “Submit form.”
•	Web Search: “Search Google for [query],” then “Read the first result.”
•	Assistant Chat: “Who is [person]?” or general questions (the extension can leverage an LLM to answer factual queries or brainstorm).
•	Settings: “Set reading speed to faster,” “Change voice to female,” “Teach you my name” (so it can personalize).
Each feature should have a clear interaction flow. For example, “List my unread emails” would open Gmail (or use an already-open tab), let the content script scrape the list (sender, subject of unread messages), and then use TTS to speak each in turn (“Email 1 from Alice: Meeting tomorrow at 10am...”).
7. Memory and Personalization
Implement a memory store (e.g. chrome.storage.local or IndexedDB). Store items like user’s name, frequently contacted people, past tasks, preferred tone of responses, etc. When conversing, pass relevant memory to the AI model as context: e.g. “User’s name is Alice, she prefers formal tone.” This allows the assistant to maintain continuity (“As I said earlier, your next meeting is on Tuesday”). For technical memory, use techniques like:
- Short-term buffer: keep the last few user commands and assistant replies in memory to maintain context within a session.
- Long-term knowledge: after each session, review important points (e.g. “The user said they like daily summaries”) and add to a persistent store.
- Use an LLM or vector embeddings to encode memory for retrieval (a RAG approach) so that very old context can be compactly stored and retrieved when needed.
By 2026, tools like LangChain or similar frameworks have mature memory modules. Incorporate a memory “module” that triggers on significant user statements (“Remember that my dog’s name is Coco”) and retrieval on queries (“What’s my dog’s name?”).
8. Implementation Roadmap
1.	Research & Design (Weeks 1-2):
2.	Finalize requirements and scope (which features for MVP).
3.	Review Chrome extension sample projects and voice accessibility extensions.
4.	Choose tech stack (e.g. pure JS vs. React for popups/UI vs. background-only).
5.	Decide AI backend (cloud API vs. local model).
6.	Extension Scaffold (Weeks 3-4):
7.	Create a basic extension with manifest v3, background and content scripts, and permissions (microphone, tabs, storage).
8.	Implement a simple “Hello world” voice trigger using SpeechRecognition and chrome.tts. Test on-device speech working as per Chrome 139+[1].
9.	Set up a basic UI or browser action icon to start/stop listening.
10.	Voice Command Engine (Weeks 5-7):
11.	Develop a parser that takes transcribed text and identifies commands (initially via simple keyword matching).
12.	Use the AI model (GPT or open-source) to refine command understanding: e.g. send “Schedule a meeting next week” to the LLM to decide it means “Open calendar and create event.”
13.	Test navigation commands (open/close tab, scroll, click).
14.	Content Reading (Weeks 8-10):
15.	Implement content script functions to extract readable text: skip navigation bars/ads, focus on main content. Possibly use the Accessibility Tree or ARIA roles.
16.	Use chrome.tts to read extracted text, with options for pausing or jumping sections.
17.	Test on various sites (news article, blog, documentation).
18.	Email Integration (Weeks 11-14):
19.	Inject into Gmail web interface: find inbox list, unread markers, open/close threads. Alternatively, integrate Gmail API (OAuth needed) for a more reliable data layer.
20.	Support voice commands for reading emails. Use TTS to read subjects and bodies, and voice to compose replies. Possibly integrate with the AI model to help draft email text (e.g. “Draft a thank-you email for meeting on [date]”).
21.	Documents/Sheets (Weeks 15-18):
22.	Similar injection into Google Docs/Sheets (or Office365 if needed): use their scripting API or manipulate content.
23.	Recognize typed vs. spoken input. Possibly convert speech to text and insert into document (like dictation).
24.	Use model to format or edit: “Make this paragraph bullet points.”
25.	Vision/AI Integration (Weeks 19-22):
26.	Decide on a vision-model approach: either connect to an API (OpenAI, Azure) or run an open-source model (via a local server or WebAssembly/TensorFlow.js).
27.	Implement a “screenshot & analyze” command: the extension takes a screenshot (tabs.captureVisibleTab) and sends it to the VLM, which returns descriptions.
28.	Use this to answer queries about images or complex layouts (tables, charts).
29.	Memory & Personalization (Weeks 23-26):
30.	Design the memory data schema (user info, conversation log, preferences).
31.	Hook up the LLM prompts to include relevant memory. For example, “The user asked about [topic] earlier, and now asks...”.
32.	Add settings UI for voice preferences, name, etc. Store these in memory.
33.	Security & Polishing (Weeks 27-30):
34.	Review all permissions; remove any not strictly needed. Add warnings to user about microphone use, data usage.
35.	Implement encryption for any stored credentials or sensitive memory. Ensure no sensitive data is logged.
36.	Perform security review: try to break auto-login, test on device vs server data flows.
37.	Testing & Iteration (Ongoing):
38.	Conduct usability testing with visually impaired users. Get feedback on voice command naturalness, reading clarity, and missing features.
39.	Benchmark performance (latency of voice commands, AI response time). Optimize (e.g. enable local recognition for speed[10], or pre-fetch data for faster responses).
40.	Iterate on conversational design: add or refine commands, improve error messages, expand AI prompts based on edge cases.
9. Technology Stack & Tools
•	Extension Platform: JavaScript/TypeScript, Manifest V3. Possibly use a framework like React or Svelte for any pop-up UI, but core is content scripts and background.
•	Web APIs: Web Speech API (SpeechRecognition, SpeechSynthesis), Chrome TTS API (chrome.tts.speak()), tabs, scripting for injection, storage for memory. Use navigator.credentials for password autofill (triggering Chrome’s account chooser)[9].
•	AI / LLM Backend: Options include:
•	OpenAI API: e.g. GPT-4/4o for general language + image understanding[5]. Pros: state-of-art, easy to use. Cons: privacy (data to cloud), cost per call.
•	Open-Source Models: e.g. Meta’s Llama 3, Google’s Gemma 3, Alibaba’s Qwen-VL, Microsoft’s Phi-4V. Can run via Hugging Face or local inference. These provide strong vision and language capabilities[11][7]. Running them locally (with WebGPU or a small dedicated server) keeps data in-house.
•	Hybrid: Use a small on-device model for common tasks (like GPT-4o-Flash-size or Gemma 1B) for quick responses, and escalate to a larger remote model if needed.
•	Data Storage: Chrome Storage (sync and local), or IndexedDB for larger data. Possibly a small on-device database (SQLite via WASM) for conversation logs.
•	Speech Recognition Library: The built-in Web Speech API is fine for Chrome. For fallback or more languages, could integrate WebAssembly versions of Whisper or similar for offline recognition (though Chrome’s on-device should cover major languages[1]).
•	Testing Tools: Use Chrome’s automated testing (e.g. Selenium or Puppeteer) to simulate voice commands (using text injection) and verify actions. Also use accessibility linting tools and screen reader (NVDA/JAWS) with speech-to-text to ensure compatibility.
10. Testing and Quality Assurance
•	Accessibility Compliance: Ensure the extension itself is usable by keyboard or screen reader (for any UI). Use WCAG guidelines to verify readability and focus management.
•	User Testing: Recruit blind or low-vision users to test actual navigation tasks (e.g. “Find and read the latest news headline”). Gather feedback on misrecognitions, false commands, or missing capabilities.
•	Performance Testing: Measure voice-recognition latency (on-device vs cloud) and TTS speed. Profile any calls to AI to avoid too long wait times – consider streaming partial responses if available.
•	Security Audit: Have an independent review of password handling and data storage. Ensure no sensitive data is logged or exposed. Check that all external communications (e.g. to AI API) use encryption.
•	Iterate on Dialogues: Use transcripts of voice interactions to identify common misunderstandings. Retrain any intent parser or refine system prompts accordingly.
